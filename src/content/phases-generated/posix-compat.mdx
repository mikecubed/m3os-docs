## Why this phase matters

Implement enough Linux-compatible syscalls and bundle musl libc so that ordinary C programs compiled on the host can run unmodified inside the OS.

## How it works in m3OS

The roadmap entry defines the learning scope for this phase; a deeper implementation doc has not been matched yet.

## How it builds on earlier phases

This phase builds on Process Model and turns that foundation into POSIX Compat.

## Implementation outline

- Audit which syscall numbers Linux assigns to the ~40 musl-required calls.
- Add a second dispatch table in the syscall gate that maps Linux numbers to internal kernel functions. The existing custom ABI remains untouched.
- Compile musl on the host with `--target x86_64-unknown-none`, patching only the syscall wrapper stubs to use `syscall` with Linux numbers.
- Write a minimal `crt0.s` that satisfies the System V entry convention.
- Bundle musl headers and the compiled `libc.a` in the disk image.
- Validate with a "hello world" C binary: compile on host, copy to image, run inside OS.

## Documentation targets

- document the Linux syscall number mapping table and dispatch strategy
- explain what musl needs vs. what glibc needs and why musl is the right choice
- document the C runtime entry sequence (`_start` → `__libc_start_main` → `main`)
- explain which syscalls are stubbed, which are real, and what the gaps mean

## Deferred follow-on work

- `futex` and pthreads
- `epoll` / `poll` / `select`
- signal delivery through the Linux signal ABI
- `/proc` filesystem entries
- dynamic linker support (`PT_INTERP`, `LD_LIBRARY_PATH`)
- `mprotect` and memory permission changes
