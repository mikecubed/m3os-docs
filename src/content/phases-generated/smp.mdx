## Why this phase matters

Boot all available CPU cores, give each its own Local APIC timer, and run the scheduler across all cores simultaneously. Two processes should be able to execute truly in parallel.

## How it works in m3OS

Phase 25 adds multi-core support to m3OS. All Application Processors (APs) discovered in the ACPI MADT are brought online via the INIT-SIPI IPI sequence. Each core has its own GDT, TSS, kernel stack, double-fault stack, LAPIC timer, and scheduler loop. Inter-processor interrupts (IPIs) provide reschedule and TLB shootdown mechanisms.

## How it builds on earlier phases

This phase builds on Persistent Storage and turns that foundation into SMP.

## Implementation outline

- Determine AP count and APIC IDs from the MADT parsed in Phase 15.
- Allocate a 4 KB trampoline page at a sub-1 MB physical address; write the 16-bit startup stub.
- For each AP: send INIT IPI, wait 10 ms, send two SIPI IPIs with the trampoline physical page number.
- In the trampoline: enable protected mode, set up a temporary GDT, enable long mode, jump to a Rust `ap_entry` function.
- In `ap_entry`: initialize the per-core GDT, TSS, LAPIC, and kernel stack; signal the BSP that this AP is alive; enter the scheduler loop.
- Split the global run queue into per-core queues; add a work-stealing path.
- Implement `send_ipi` using LAPIC ICR registers.
- Add TLB shootdown IPI handler.
