## Why this phase matters

m3OS supports a small, deliberate reference-hardware matrix with a documented driver strategy, a reusable hardware-access layer, and at least one serious real-hardware storage and networking path. The project stops talking about hardware support as a vague future aspiration and starts treating it as a bounded support promise.

## How it works in m3OS

The roadmap entry defines the learning scope for this phase; a deeper implementation doc has not been matched yet.

## How it builds on earlier phases

This phase builds on Post Serverization Kernel Hygiene and turns that foundation into Hardware Substrate.

## Implementation outline

- Choose the initial reference hardware matrix and driver targets.
- Define the donor strategy and document it in the roadmap and hardware docs.
- Implement the hardware-access layer for BAR mapping, DMA, IRQs, and device binding.
- Bring up the first serious storage and networking drivers on the reference targets.
- Add reproducible validation steps for real-hardware bring-up.
- Update support-matrix and subsystem docs to reflect the new hardware posture.
