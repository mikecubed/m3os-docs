## Why this phase matters

All CPU cores dispatch and run user tasks. The BSP-only restriction is removed by making syscall infrastructure per-core. The scheduler uses per-CPU run queues with load balancing, and tasks have priority levels. This is the single highest-impact performance improvement — the OS goes from effectively single-core to true multi-core.

## How it works in m3OS

The roadmap entry defines the learning scope for this phase; a deeper implementation doc has not been matched yet.

## How it builds on earlier phases

This phase builds on Real Time Clock and turns that foundation into True SMP Multitasking.

## Implementation outline

- Move syscall statics into `PerCoreData` and update `syscall_entry` assembly.
- Verify: two cores can handle syscalls simultaneously without corruption.
- Remove BSP-only guard in `pick_next()`.
- Verify: tasks run on multiple cores (log which core runs which task).
- Implement per-CPU run queues, replacing the global scheduler lock.
- Implement load balancing (periodic migration).
- Add priority field to `Task`; update `pick_next()` to respect priorities.
- Implement `sched_setaffinity` / `sched_getaffinity` syscalls.
- Implement wait queues; attach to pipes and sockets.
- Add time accounting; implement `times()` syscall.
- Stress test with many concurrent processes across all cores.
