## Why this phase matters

m3OS can run DOOM as a real full-screen graphical application. The game loads its WAD data from disk, renders through the framebuffer path, accepts keyboard input for gameplay, and returns the system to a usable administration path when it exits.

## How it works in m3OS

Phase 47 is where m3OS stops proving isolated subsystems and starts carrying a real graphical application end to end. It adds framebuffer mapping, raw scancode input, doomgeneric build integration, shareware WAD delivery on the ext2 image, and the debugging work needed to keep a real game responsive under QEMU. The end result is a working DOOM port with known remaining performance headroom.

## How it builds on earlier phases

This phase builds on System Services and turns that foundation into Doom.

## Implementation outline

- Define the documented userspace framebuffer and keyboard path used by the port.
- Implement or finish the platform layer for DOOM.
- Integrate the WAD and binary into the supported image/build flow.
- Validate launch, rendering, gameplay input, and exit/recovery behavior.
- Document what this phase proves and what it does not prove about the later GUI stack.
