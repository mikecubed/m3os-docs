## Why this phase matters

DOOM runs inside the OS. The shareware `doom1.wad` loads from disk, renders to the framebuffer, and accepts keyboard input for gameplay. This is the "it runs DOOM" milestone — the classic proof that a hobby OS has reached a meaningful level of capability.

## How it works in m3OS

The roadmap entry defines the learning scope for this phase; a deeper implementation doc has not been matched yet.

## How it builds on earlier phases

This phase builds on System Services and turns that foundation into Doom.

## Implementation outline

- Implement `sys_framebuffer_info` syscall — return dimensions, pitch, pixel format.
- Implement `sys_framebuffer_mmap` syscall — map framebuffer into userspace.
- Implement `sys_read_scancode` syscall — return raw PS/2 scancodes.
- Clone doomgeneric source; write m3os platform layer (~200 lines).
- Cross-compile with `musl-gcc -static`; add to xtask build.
- Add `doom1.wad` to the ext2 disk image.
- Boot with `cargo xtask run-gui` and test.
- Tune scaling and input mapping.
- Screenshot the running game for the README.
