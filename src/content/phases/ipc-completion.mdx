## Why this phase matters

m3OS gains the missing IPC and bulk-data pieces required for serious userspace services: safe capability transfer, practical large-payload transport, ring-3-safe service registration, and clear failure semantics for the canonical server loop.

## How it works in m3OS

Phase 50 finishes the IPC transport model that Phase 6 introduced as a control-path-only design. The kernel now supports safe capability transfer between process capability tables, a two-tier bulk-data mechanism (validated copy for small payloads, page grants for large transfers), ring-3-safe service registration with owner tracking and re-registration, and documented server-loop failure semantics. These additions close the gap between the existing message-passing infrastructure and the requirements of real ring-3 services.

## How it builds on earlier phases

This phase builds on Architectural Declaration and turns that foundation into IPC Completion.

## Implementation outline

- Audit current IPC shortcuts and identify every service path that still depends on shared kernel addresses.
- Finish and document the capability-grant semantics.
- Implement the generic bulk-data transport model and its ownership rules.
- Remove kernel-only assumptions from the service registry and service-address paths.
- Standardize the canonical server loop and service-death semantics.
- Port at least one real service path to the new transport as proof.
- Extend validation so later extraction phases can depend on this transport with confidence.
