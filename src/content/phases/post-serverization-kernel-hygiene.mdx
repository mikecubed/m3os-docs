## Why this phase matters

Close the `FdEntry` CLOEXEC / NONBLOCK plumbing gap and relocate the four `arch::x86_64::syscall::*_pub` wrappers into their owning subsystems so later phases do not inherit a silent security bug or a process-module arch dependency.

## How it works in m3OS

Phase 54 moves meaningful storage, pathname, and networking policy out of ring 0 without breaking the existing Linux-like syscall ABI. The kernel keeps the object handles, transport, and bootstrap fallback paths; supervised userspace services now own the migrated policy slices:
The result is not a fully serverized OS, but it is a materially narrower kernel than the Phase 53 baseline and a more honest microkernel story.

## How it builds on earlier phases

This phase builds on Deep Serverization and turns that foundation into Post Serverization Kernel Hygiene.

## Implementation outline

- Introduce `FdEntry::from_open_flags(backend, flags)` in the module that owns `FdEntry`.
- Convert every hardcoded `cloexec: false, nonblock: false` construction site in `kernel/src/arch/x86_64/syscall`, including `vfs_service_open`.
- Add a per-backend userspace regression test that execs and verifies `O_CLOEXEC` cleared the fd.
- Relocate each of the four `*_pub` wrappers into its owning subsystem; update `kernel/src/process/mod.rs` call sites.
- Trim `docs/debug/54-followups.md` to the two long-term backlog items (MOUNT_OP_LOCK yielding, scheduler thresholds) that intentionally remain open.
