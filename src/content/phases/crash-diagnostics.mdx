## Why this phase matters

Every kernel crash is self-diagnosing. When the kernel panics or takes a fatal fault, the serial output includes full CPU register state, current task metadata, and per-core scheduler context. `debug_assert!` invariants at every scheduler, fork, IPC, and stack boundary catch state corruption at the point of origin rather than surfacing later as an opaque `RIP=0x4` fault.

## How it works in m3OS

Phase 43a makes every kernel crash self-diagnosing. The panic handler and all fault handlers now print full CPU register state, current task metadata, and per-core scheduler context. `debug_assert!` invariants at scheduler, fork, IPC, and stack boundaries catch state corruption at the point of origin rather than surfacing later as an opaque fault.

## How it builds on earlier phases

This phase builds on SSH Server and turns that foundation into Crash Diagnostics.

## Implementation outline

- Create `kernel/src/panic_diag.rs` with `dump_crash_context()` (registers, task info, per-core state).
- Update the panic handler in `kernel/src/main.rs` to call `dump_crash_context()`.
- Add `try_lock_scheduler()` and `Scheduler::get_task()` for safe non-blocking scheduler access.
- Enrich page fault, GPF, and double fault handlers to call `dump_crash_context()` with additional context.
- Add `debug_assert!` checks in `pick_next`, `yield_now`, `block_current`, `wake_task`, `run`, and `enqueue_to_core`.
- Add `debug_assert!` checks in `make_fork_ctx`, `fork_child_trampoline`, `sys_fork`, and `spawn_fork_task`.
- Add `debug_assert!` checks in `recv_msg`, `send`, and `reply`.
- Add `Task::stack_bounds()` and stack validation assertions in `run`.
- Verify `cargo xtask check` and `cargo xtask test` pass.
