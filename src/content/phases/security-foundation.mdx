## Why this phase matters

m3OS closes the trust-floor gaps that currently block any honest claim of "safe enough for serious headless use." When this phase lands, identity transitions, password storage, entropy, and default remote-exposure behavior all match the project's intended multi-user and network-facing story.

## How it works in m3OS

Phase 48 closes the trust-floor gaps in the m3OS multi-user and network-facing security model. It replaces advisory, demo-quality defaults with real enforcement: kernel-checked credential transitions, hardware-backed entropy, iterated password hashing, and safe boot defaults.

## How it builds on earlier phases

This phase builds on Doom and turns that foundation into Security Foundation.

## Implementation outline

- Audit the current trust-floor failures and map each one to a concrete kernel, userspace, or image-default fix.
- Implement kernel-side credential transition enforcement and remove any remaining trust shortcuts.
- Replace the current weak entropy path with a documented stronger CSPRNG/seed pipeline.
- Remove baked-in default credentials and define the replacement provisioning or first-boot flow.
- Make telnet opt-in only and align init/service defaults with the new exposure policy.
- Upgrade password hashing and account-file update semantics.
- Add or extend smoke/regression coverage for login, SSH, password changes, and boot defaults.
- Update documentation, release framing, and version references for the new trust floor.
