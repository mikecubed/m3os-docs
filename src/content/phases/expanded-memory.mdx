## Why this phase matters

The kernel supports demand paging, large `mmap()` regions, and `mprotect()`. Virtual pages are mapped lazily — physical frames are allocated on first access via the page fault handler. This unlocks running large cross-compiled binaries (Clang, Python, Node.js, git) that allocate hundreds of megabytes via `mmap()` but only touch a fraction of the mapped pages. QEMU RAM is increased to support these workloads.

## Implementation outline

- Extend `MemoryMapping` with `prot` and `flags` fields.
- Change `sys_linux_mmap()` to record the VMA without allocating frames.
- Extend the page fault handler to check the VMA list and demand-map valid lazy faults.
- Implement `mprotect()` syscall with page table walks and TLB shootdown.
- Increase QEMU RAM to 1 GB.
- Expand data partition to 1 GB.
- Validate CoW fork still works alongside demand paging.
- Test with large mmap regions (256 MB+) and a cross-compiled binary.
