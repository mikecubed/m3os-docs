## Why this phase exists

Every later subsystem depends on a trustworthy boot path. Before memory management, interrupts, or userspace exist, the system needs one thing above all else: a dependable way to prove that control reached the kernel and to see what happened next.

## How it works in m3OS

The host-side `xtask` helper builds the kernel, wraps it in a UEFI image, and launches QEMU with OVMF firmware. The bootloader reaches `kernel_main`, the serial backend initializes, and the kernel emits a readable startup message before idling.

## How it builds forward

This phase creates the diagnostic baseline that every later phase depends on. Memory setup, interrupt bring-up, and tasking are all much easier to debug because the serial path already exists.
