## Why this phase matters

Move from a static early kernel into a kernel that can allocate memory safely and reason about physical and virtual address space.

## How it works in m3OS

After Phase 1, the kernel could boot and print diagnostics, but it still had no way to manage RAM dynamically. Phase 2 adds the minimum memory machinery needed to build future subsystems:

## How it builds on earlier phases

This phase builds on Boot Foundation and turns that foundation into Memory Basics.

## Implementation outline

- Store a `'static` reference to the bootloader memory map (the regions slice is valid for the kernel's lifetime) and parse it into typed kernel structures.
- Implement a simple frame allocator with conservative rules.
- Expose page-table operations through safe wrappers around `x86_64`.
- Map a small heap and initialize the allocator.
- Add simple allocation tests and logging for confidence.
