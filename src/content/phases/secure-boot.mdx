realOsDifferences:
  - "Linux distributions use a **shim** first-stage bootloader signed by Microsoft's UEFI CA. The shim maintains its own **MOK (Machine Owner Key)** database and verifies GRUB, which in turn verifies the kernel. This lets distros ship updates without involving Microsoft on every kernel release."
  - "The shim approach requires applying to the Microsoft UEFI CA program (months, legal entity required), signing the shim binary with Microsoft's key, and then using the shim to trust the distro's own key for subsequent stages."
  - "For a personal OS there are two practical options:"
  - "**Via shim's MOK** (`mokutil --import`): the easiest path on machines that already boot through shim. The key lives in shim's MOK list, not the UEFI firmware db."
  - "**Direct UEFI db enrollment** (`efi-updatevar` or firmware setup): adds the key to the firmware's own signature database, works without shim, but requires putting the firmware into Setup Mode and re-enrolling the PK/KEK hierarchy."
  - "Windows uses a similar hierarchy but its keys are enrolled by the OEM at manufacturing time. End users cannot easily add their own keys on consumer hardware that ships with \"Windows Secure Boot\" locked down (though most firmware allows it in the UEFI setup)."

## Why this phase matters

Sign the kernel's EFI binary so it can boot on real hardware with UEFI Secure Boot enabled, without needing to disable the firmware security check.

## Implementation outline

- Generate a 4096-bit RSA key pair and self-signed X.509 certificate with `openssl`.
- Add `cargo xtask sign <unsigned-efi>` for signing any EFI executable with optional `--key` / `--cert` overrides.
- Extend `cargo xtask image` with `--sign` so the build produces a signed bootable disk image in addition to the unsigned image.
- Document the one-time certificate enrollment. Two paths exist — choose based on the target machine (see above).
- Verify the signed binary boots with `mokutil --sb-state` or `dmesg` reporting Secure Boot enabled.
